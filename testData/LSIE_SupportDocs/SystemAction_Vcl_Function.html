Overview¶

SystemAction and ModuleAction are built-in functions that can be triggered to perform additional module actions within Solution.Demo. Using these functions avoids the necessity of working with module management code.

Replacing VCL with these functions helps to reduce duplicate code throughout your solution. Because SystemAction and ModuleAction are built-in and do not require any VCL construction, it is not possible to break a module when upgrading your solution. Solutions working in iTest 3.4 are usable by future versions of iTest without having to make any additional module changes.

Built-In Actions and States¶

The SystemAction VCL function (described in detail further down the page) takes the following three arguments:

+0.0 SET retval SystemAction("INIT", flgActionComplete, "ALL")

The first argument is a string constant or string channel, and it defines the action that you wish to trigger. In this case, the action to trigger is INIT or "Initialize." The second argument is the virtual output channel that will get set to "1" by the SystemAction function when the action is complete. The third argument is a string that represents which modules the SystemAction function will send this action to. In this case, all modules are to be sent the INIT action, no matter what their current state.

Each built-in action has a start state and an end state. These start and end states have both a channel value (NumSysStatus) and a string channel label (VsSysStatus) that gets set by the SystemAction function when an action is triggered. For example, the start state for the INIT action has a NumSysStatus value of 3 and a VsSysStatus label of "Initializing."

Using the example above, SystemAction will trigger the INIT action for all modules. Next, it will update the NumSysStatus system channel for every module to "3" and the VsSysStatus system channel to "Initializing" for the start state, and then to "4" and "Initialized" when the INIT action is complete.

The default built-in actions, along with their start and end state values, are summarized in the following table:

Built-in Action Details

Action Start State End State Description
INIT 3 = Initializing 4 = Initialized Started on launch and after test builder completes.
STOP 5 = Stopping 6 = Stopped Test cell stop.
READY 7 = Getting ready 8 = Ready Prepare test cell and equipment for test.
RESET 9 = Resetting 10 = Reset Reset safeties.
START 11 = Starting 12 = Started Start the engine (or UUT).
IDLE 13 = Idling 14 = Idle Idle the engine (or UUT idle).
EXIT 15 = Exiting 16 = Exit iTest is exiting; shutdown the module.
ENGINESTOP 17 = Stopping 8 = Ready Stopping the engine (or UUT).
RESETFAULTS None None Acknowledge or clear faults.

 

Most of the built-in actions also have conditional checks, or state checks, that need to be met before the SystemAction function can execute the action. The INIT action has no required check. It does not matter what state the module was in before the action was sent; it automatically sets the NumSysStatus channel value to 3 and starts initializing. For the READY action, however, the requirement is that the NumSysStatus channel value must be set at less than 4 first, which means that the INIT action must first be run to completion (3 then 4) before READY can be triggered.

If an action is sent and a module's channel value is not in the required state, you will be notified by a system fault (triggered by the application code).

The table below summarizes the required state checks for the default actions:

State Checks

Action State Check
INIT None
STOP None
READY numSysStatus < 4
RESET numSysStatus >= 10
START numSysStatus >= 12
IDLE numSysStatus < 12
numSysStatus >= 14
EXIT numSysStatus > 6
ENGINESTOP numSysStatus < 9

 

These actions are guaranteed to be supported by iTest. The numeric values of the actions and the state transitions are consistent, and the configurability of these actions is minimal.

Custom Actions and the XML Configuration File¶

  • The name of the file is SystemActions.xml, and it should be in the solution folder. No GUI editor exists.
  • It is used to customize the behavior of the built-in actions or add new actions and states.
  • Custom actions are defined in the XML file.
  • The XML file has the following sections:
<Actions>
    <Action>
        <Name>MyCustomAction1</Name>
        <Timeout>60</Timeout>
        <StartState>18</StartState>
        <Logging>Verbose</Logging>
        <StartStateText>Custom1Starting</StartStateText>
        <EndState>19</EndState>
        <EndStateText>Custom1Started</EndStateText>
                <CheckBusyFlag>FALSE</CheckBusyFlag>
        <StateChecks>
            <StateCheck>
                <Equation><![CDATA[numSysStatus < 4]]></Equation>
                <Text>Must not be less than 4</Text>
            </StateCheck>
                <StateCheck>
                       <Equation>numSysStatus &gt;= 8</Equation>
               </StateCheck>
           </StateChecks>
    </Action>
    <Action>
        <Name>MyCustomAction2</Name>
            .
            .
            .
    </Action>
</Actions>
  • Due to XML syntax, the characters <, >, ', ", and & cannot be used directly in the text sections. They must be escaped by one of two methods. See the above sample for examples.
  • The following table lists the escape codes for the characters:

Character Escape Codes

Character Escape Code
< &lt;
> &gt;
& &amp;
' &apos;
" &quot;
  • The name can be one of INIT, READY, RESET, START, STOP, IDLE, or EXIT to override the behavior of one of the built-ins, or it can be the name of a custom state. A custom state cannot have any spaces.
  • The timeout value is the number of seconds SystemAction waits before continuing. The minimum timeout value for SystemAction is 1.
  • The Start State is set when the action begins (it sets the vsSysStatus and numSysStatus channels).
  • The End State is set if the action finishes correctly.
  • Setting logging to Verbose gets more Softpanel messages. Leaving this setting out results in a minimum of Softpanel messages during execution of an action.
  • New Start State and New End State values and strings cannot overlap with another action.
  • Start State and End State must be consecutive numerically.
  • Start State and End State text labels can use the language macros from the language file. See iTest Language File.
  • CheckBusyFlag is used to turn off checking the busy flag for that custom action. This means that this action can run in parallel with other actions. Built-in actions are mutually exclusive. However, you can design custom actions to execute independently. It is recommended that custom actions (which do not check the busy flag) also not include Start State and End State numbers or text because they could then overwrite the Start State and End State values being written by a built-in action.
  • Start State and End State cannot overlap with a built-in action; therefore, the first number that can be used is 18.
  • Custom actions do not need to provide state values and state text. If it does not provide these, then vsSysStatus and numSysStatus are not updated during those actions.
  • StateCheck is a series of equations that is checked for true before the action occurs. If any are true, the action aborts. The StateCheck syntax is operand operator operand. 
     numSystatus > 4
  • The 7 built-in actions can only have StateText, Logging, Timeout, and StateChecks configured from the XML. All other settings will be ignored for built-ins.
  • The advantage of using custom actions instead of writing new actions in VCL is that you do not have to worry about module communication and handling. All of that is already managed.

SystemAction VCL Function¶

Syntax¶

The SystemAction VCL function uses the following syntax:

+0.0 SET retval SystemAction(<Action>,<VirtualOutputChannel>,<Module>)

Argument Details:

<Action>: The first argument can be a string constant or a string channel. The value of this channel should be a built-in action such as, "INIT", "READY", "RESET", "START", "STOP", "IDLE", or "EXIT". It can also be the name of a custom action defined in SystemActions.xml.

<VirtualOutputChannel>: The second argument is a virtual output channel that is set to 1 when the action is complete.

<Module>: The third argument is a string that represents the module that the action is sent to. If the module string is "ALL" or empty (""), then all modules receive the action. This string can also represent multiple module IDs by specifying a substring (e.g., "Eb"), which would call all modules with "Eb" in the module ID.

Example:

+0.0 SET retval SystemAction("INIT",flgActionComplete,"ALL")

Return Values:

System Action Return Values

Value Description
0 Action executed successfully.
-1 SystemActions.xml <statecheck> evaluates to true.
-2 Action name not found in SystemActions.xml.
-3 System busy (channel: flgSysBusy=1).
-4 Action already requested and currently in-progress.

 

Logic¶

SystemAction Logic is as follows:

  • Check the flgSysBusy flag. If it is not zero, then exit. The STOP, ENGINESTOP, and RESETFAULTS actions do not check flgSysBusy.
  • Check the StateChecks to see if you can execute this action. StateChecks comes from the XML file, but include the following:
    • EXIT will not start if flgSysRunning = 1.
  • Set flgSysBusy = 1.
  • Set Begin State channels (vsSysStatus and numSysStatus).
  • For READY, RESET, and INIT states, module fault flags and flgSysFault will be reset to 0.
  • Call the SysPre Action VCL function, where the action is Idle, Reset, Stop, etc. (if it exists).
NOTE: If the vsSysSafetyModule virtual string channel is not empty for the RESET built-in action and contains a valid module ID, then that module's reset action is called before calling all of the other module reset actions.
  • For each module that is not disabled (flgMIDDisabled = 1), call the MID Action module function (where Action is Init, Ready, etc.) if it exists. If the module does not have that function defined (and it is a built-in action), then call the MID Action module function with the numSysAction channel set to:
    • 1 (INIT)
    • 2 (READY)
    • 3 (RESET)
    • 4 (START)
    • 5 (IDLE)
    • 6 (STOP)
    • 7 (EXIT)
    • 11 (ENGINESTOP)
  • Check for faults (vsSysStatus=Fault); exit action on fault unless the action is Init.
    • On exit, final values for numSysStatus or vsSysStatus are not set.
    • The "Fault" setting for vsSysStatus is not case sensitive.
  • If there is no fault or if the action is Init, call the SysPost Action VCL function, where the action is Idle, Reset, Stop, etc. (if it exists).
  • For STOP and EXIT states, mailslot messages are sent to Data_Engine to save module INI files and Sys.ini. Saving the INI files has a hard-coded timeout of 5 seconds. This event can be turned on and off via the WriteSysIniChannel powertek.ini setting.
  • Set End State if successful.
  • Set flgSysBusy = 0 before exiting.

System Channels¶

The SystemAction function makes sure that certain channels are available in every solution. If they are not there, they are added. SystemAction is responsible for updating or reading the following channel values during the execution of a system action:

  • flgSysBusy - Set to 1 when an action begins and 0 when it is complete.
  • flgSysRunning - This is an indication that the system is running. It is used as a built-in StateCheck for the EXIT Action.
  • numSysModules - Total number of modules in the system.
  • vsSysStatus - String state value set at the start and end of each action.
  • numSysStatus - Numeric state value set at the start and end of each action.
  • tmrSysStatus - Timer that shows how long you have been in each action.
  • vsSysSafetyModule - Set to the Module ID of the safety module.
  • flgSysIniWrite - Used by SystemAction() to determine when module INI write commands have finished.
  • flgSysFault - Overall fault flag for the system.

From the modules, SystemAction is looking for the following channels. Note that it does not add them automatically if they are not present.

  • flgMIDFault - Did this module cause a fault?
  • flgMIDDisabled - Is this module disabled?
  • flgMIDBusy - Busy channel from each module. Initially set to 1 by the system; set to 0 by module when the action is complete. If this channel is not present, then SystemAction will be unable to monitor for the completion of the action from that module. The module will be assumed to be complete with the action immediately.

The following channels and VCL procedures have been removed from A&D standard solutions as this framework makes them obsolete:

  • flgSysModuleStatus
  • vsSysModuleStatusChannel
  • vsSysProcedureName
  • SysAction.TSC
  • SysSchModuleAction.TSC
  • numSysModule - This should be moved to a local variable in the places where it is still needed. It is only used as a loop counter.

Constraints¶

There are a maximum of 50 actions that can be defined in a solution. This number includes the built-ins, so a maximum of 42 custom actions can be supported.

ModuleAction¶

There is an occasional requirement that a module be able to call a function in another module without having an explicit reference to a specific module's function name. To do this, use the ModuleAction function.

ModuleAction has the exact same syntax and nearly the same functionality as SystemAction. The syntax is as follows:

  • It takes 3 arguments - ModuleAction(“action”, flgCompletionFlag, "modules")
    +0.0 SET retval ModuleAction("Purge", flgActionComplete, "Eb")
  • The first argument will be either a string constant or string channel. The value should be a string that will evaluate to a procedure when the module ID is prepended to the string. For example, "Purge" in the example above would become Eb1Purge or Eb2Purge. At runtime, iTest will try to call those procedure names.
  • The second argument is a VO channel which will get set 0 when the action is started and to 1 when the action is complete.
  • The third argument is a string that represents the modules to send the action to. If the string is "ALL" or empty (""), then all modules receive the action. If the string is a specific module ID, then only that module will receive the action. Alternatively, the string could represent multiple module IDs by specifying a substring like "Eb" which would call all modules having "Eb" in their module ID.
  • The return value will be set to -1 if the action is already in process or if the maximum number (10) of actions that are already "in process" has been reached. A positive return value indicates the number of modules that had a matching VCL function for the given action string. A 0 indicates that no functions were found that matched.

If an action is blocked from running, an error will occur stating that the requested action is blocked; this error can be viewed via the System Messages tab in the Messages panel.

A requested action will be blocked from running under the following conditions:
  • The maximum number (10) of actions are already running.
  • The requested module(s) exactly match a running action's module(s).
    • E.g., the "Eb1" module is requested, but there is already a running action for the "Eb1" module.
    • E.g., the "Eb" modules are requested, but there is already a running action for the "Eb" modules.
    • E.g., all modules are requested, but there is already a running action for all modules.
  • The requested module(s) would be a subset of the running actions' module(s).
    • E.g., the "Eb1" module is requested and there is already a running action for all modules or for the "Eb" modules.
  • The requested module(s) would include a running action's module(s) as a subset.
    • E.g., all modules or the "Eb" modules are requested when there is already an action running for the "Eb1" module.

Differences Between ModuleAction and SystemAction¶

  • SystemAction will only accept pre-built action names or names that were predefined in the XML file.
  • SystemAction does not set the flgActionComplete channel to 0 when the action is started.
  • ModuleAction will accept any ad-hoc string.
  • ModuleAction does not set the flgSysBusy flag.
  • ModuleAction does not set the value of numSysStatus or vsSysStatus.
  • ModuleAction does not check the value of flgSysFault after each module completes its action.
  • ModuleAction always uses a 60 second timeout.
