Overview¶

Softpanel messages are processed by the main application (iTest Console or AutomationPanel). They are primarily used to change aspects of the application's display area. There are 6 types of messages:

  1. Prompter Messages: Messages used to control the usage and display of the system prompter
  2. PID Messages: Messages used to control the PID mode pair dialog
  3. Display Messages: Messages used to launch and close various displays in iTest Console
  4. Other Messages Miscellaneous messages
  5. BTS Messages: Messages used by BTS Applications
  6. AutomationPanel Messages: Commands specific to AutomationPanel

The address for Softpanel is Softpanel. All messages should reference an address such as the following:

\\.\\mailslot\Softpanel>>Your Message

 

NOTE: You can use the tilde ( ) to escape characters in mailslot messages.

 

Notes About this Document¶

The version numbers reference the product version that the message was introduced.

Prompter Messages¶

Prompter dialog boxes are queued up for user input unless otherwise noted. The title bar of the visible dialog box will list how many are queued behind the visible one.

$CHANNEL OFF¶

Turns off a dialog box initiated by $PROMPTER; $CHANNEL.

Version:
2.1
 

$PROMPTER;message¶

Displays a dialog box with the specified message (e.g., “Communication established”) and an OK button that must be acknowledged by the user. This dialog box will be replaced by the next opened $PROMPTER, instead of taking a spot in the queue for future user input.

Version:
2.1

Syntax:

$PROMPTER;Communication established

 

$PROMPTER;message;$LEVEL;messagelevel¶

Displays a dialog box with the specified message and an alternative background color. This dialog box will be replaced by the next opened $PROMPTER, instead of taking a spot in the queue for future user input.

Message level colors:

1 = Green
2 = Yellow
3 = Red

Version:
2.1

Syntax:

$PROMPTER; An Error has occurred;$LEVEL;3

 

$PROMPTER;message;$DBSTRING;channelname¶

Displays a dialog box with a text field that allows entry of a string that is then populated to a virtual string channel.

Version:
2.1
 

$PROMPTER;message;$CHANNEL;channelname;buttonname1;buttonname2¶

Returns a zero or one to a channel. Selecting button name 1 sets the channel to 1; selecting button name 2 sets the channel to 0. Typically, this type of prompt is used from Autotests to let the user control sequencing of events.

Version:
2.1

Syntax:

$PROMPTER;Please Respond;CHANNEL;PASS_FAIL;PASS;FAIL

$PROMPTER;message;$X;<X>;$Y;<Y>¶

Optional parameters for the $PROMPTER mailslot in AutomationPanel:

  • $X;<X> - the distance between the left edge of the prompter window and the left edge of the screen
  • $Y;<Y> - the distance between the top edge of the prompter window and the top edge of the screen

Version:
2.1

Syntax:

$PROMPTER;message;$X;800;$Y;400

$EXECUTE¶

Executes any executable, process, or batch file in the system. The system directory name (Execute directory) is set as the working directory. Any other directory requires that the full path be defined.

NOTE: EXECUTE commands should now be sent to FileServices for more reliable handling of EXECUTE requests.

Version:
2.1

Syntax:

$EXECUTE; <File;Silent>

Arguments:

<File>: The full path of the file. If there are spaces in the file name or accompanying space-delimited arguments, double quotes must surround the entire <File;Silent> argument.

[;Silent]: Optional feature that launches the executable in hidden mode. It follows the entire <File> string, including any arguments to be sent to the launched executable.

NOTE: The ;Silent flag is supported by non-GUI applications but will vary with GUI applications depending on their internal startup processes.

 

$EXIT¶

Automatically shuts down iTest Console or AutomationPanel without prompting the user.

Version:
2.1

Version:
4.0

For AutomationPanel.exe only, support was added for two optional arguments:

PromptSave: Causes a save dialog (Yes/No/Cancel) to be displayed on exit if something was modified.

ReturnChannel=: Only used if PromptSave is specified. The specified channel will be set to 1 if Cancel was selected from the save dialog or 0 if Yes or No or no dialog was displayed.

Syntax:

$EXIT;PromptSave;ReturnChannel=voSysExitCancelled

Typically, the PID control dialog only loads its mode selections from a file called mode.lst, which is in the root of $SUPPORTDIR. This option allows you to load in unique mode files on the fly depending upon changing test conditions. The pid1mode and pid2mode options specify which mode should be visible initially when the new file is loaded in. Creating the additional mode files will need to be done using a text editor.

Version:
2.2

Syntax:

LOADMODEFILE; <filename> <pid1mode> <pid2mode>

AutomationPanel Messages¶

$CLOSEDISPLAY¶

Sends DIE message to the specified display and wait for the display to close

Version:
3.5

Syntax:

$CLOSEDISPLAY;<displayname>

 

$CLOSELAYOUT¶

Closes an open layout that matches the specified filename if it's already open. You will NOT get the normal prompt that you get when closing from the close button in the upper right.

Syntax:

$CLOSELAYOUT;layoutfilename [StatusChan=<voChan>]

Where the file extension is not included in the layoutfilename argument.

Example:

To close the layout opened from LabWorX.layout in $SUPPORTDIR\Layouts, send the following message:

$CLOSELAYOUT;LabWorX

To close the layout opened from the Main.layout in $SUPPORTDIR\Layouts and use the optional argument to store the message status to the voFlgClose channel, send the following message:

$CLOSELAYOUT;Main StatusChan=voFlgClose

Where the returned status values are as follows:

  • 1 - busy
  • 0 - success
  • -1 - error

 

$FLEXEDIT¶

Opens FlexEdit, opens FlexEdit with the specified XML file, or opens FlexEdit displaying only the editor for the specified file and nothing else (FlexEdit.xml is not used).

Version:
3.5

Syntax:

$FLEXEDIT
$FLEXEDIT;<xml file path>
$FLEXEDIT;<config file path>
$FLEXEDIT;<ini filepath>;Editable=<true or false>

When used in this way, FlexEdit can replace the old $FORM message for displaying the contents of a .UVL file. For example, calling FlexEdit with the following option will show FlexEdit with an .ini style editor for editing the data file header values at the start of a test.

$FLEXEDIT;$SUPPORTDIR\varlists\MyVars.uvl

When used to edit an INI file, the optional Editable argument can be added to allow the user to add new channels to the file. By default, the user is unable to add new channels.
 

$GetItemProperty¶

Gets the value of a property for a panel/display object and sets an iTest channel to this value. This message requires a Property ID, which is specified in Property Manager.

Syntax:

$GetItemProperty;<PropertyID>=<ChannelName>

Arguments:

<PropertyID>: The name of the Automation ID and property name. This is visible at the bottom of the Property Manager dialog when you select a property that supports automation.

<ChannelName>: The name of the iTest channel set to the property value of a panel/display object.

Property ID Syntax:

<AutomationID>.<PropertyName>

Arguments:

<AutomationID>: The name assigned to the panel or object in Property Manager.

<PropertyName>: The internal name for the property. (This is not necessarily the name displayed in the left column of Property Manager.)

NOTE: If an object doesn't have an Automation ID defined, then none of its properties can be modified, and Property Manager will not display a PropertyID.

$NOTIFY¶

Allows both system and test code to popup a transient message to the operator. Once the message appears, the operator has 7 seconds to pin or close the message before it closes on its own. After the message is closed, then the only place the message will be visible is in the message queue. If another message comes in before the last one is closed, then it will be stacked just above the previous message.

Version:
3.5

Syntax:

$NOTIFY;CAPTION=<caption text>;TEXT=<message text>;LEVEL=<level>

Where LEVEL maps to the following behaviors:

Level Default Caption Text Text Color
0 Notice Blue
1 Notice Green
2 Warning Yellow
3 Error Red
(other) Notice Black

 

The CAPTION and TEXT settings can use the HTML formatting tags from the previous section. The <color> tag will override the LEVEL setting if it encloses the caption or message text.

If CAPTION is not specified, then the LEVEL setting determines the caption text.

The $NOTIFY message also supports translation and channel substitution (with the use of macros) within the caption or message text.
 

$OPENLAYOUT¶

Shows the layout name specified in the message. You can open layouts in AutomationPanel, or you can load them outside the application as a standalone display. If the layout is already open, it will be activated.

Open in Application¶

To open the layout in the application, use the following syntax:

Syntax:

$OPENLAYOUT;<layoutname> [StatusChan=<statuschan>] [Modal=<modal>]

Arguments:

<layoutname>: The name of the layout that you want to open.

<statuschan>: (Optional) The name of the virtual output channel used to display the status of the message. If the layout is busy, the channel is set to 1. If the layout loaded successfully, the channel is set to 0. If the layout didn't load successfully, the channel is set to -1.

<modal>: (Optional) TRUE/FALSE. If TRUE, the layout will open as a modal layout. The presence of a modal layout disables interaction with the main AutomationPanel window as well as any floating layouts. Users can only interact with the modal layout until it is closed. Other layouts cannot be opened while a modal layout is open.

NOTE: Modal layouts do not suppress closing AutomationPanel via a $EXIT message.

Example:

To open the layout file, LabWorX.layout, in $SUPPORTDIR\Layouts, send the following message:

$OPENLAYOUT;LabWorX

To open the layout file, Main.layout, in $SUPPORTDIR\Layouts and use the optional argument to store the message status to the voFlgOpen channel, send the following message:

$OPENLAYOUT;Main StatusChan=voFlgOpen

To open the layout file, LabWorX.layout, in $SUPPORTDIR\Layouts as modal, send the following message:

$OPENLAYOUT;LabWorX Modal=TRUE

Open as a Standalone Display¶

To open the layout as a standalone display, use the following syntax:

Syntax:

$OPENLAYOUT;<layoutname> [StatusChan=<statuschan>];<x,y>;<w,h>;<true/false>

Arguments:

<layoutname>: The name of the layout that you want to open.

<x,y>: The location of the layout when it's opened, where x and y are coordinates starting from the upper-left corner of the screen indicating where the layout should be opened.

<w,h>: The size of the layout when it's opened, where w is width and h is height.

<true/false>: The argument that determines whether the layout is opened in a separate window using DspDrv_AP.exe or normally in AutomationPanel. To launch using DspDrv_AP.exe, set this value to true. By default, it is false.

<statuschan>: (Optional) The name of the virtual output channel used to display the status of the message. DspDrv_AP.exe sets the channel to 1 before the layout is loaded. If the layout loaded successfully, the channel is set to 0. If the layout didn't load successfully, the channel is set to -1.

Example:

$OPENLAYOUT;Main;10,10;200,200;true

In the example above, Main.layout will open at location 10,10 and size 200x200 as a standalone layout.

NOTE: Modal layouts are not supported by standalone syntax.

$ProgressBar¶

Displays a time-based progress bar dialog.

Version:
3.8

Syntax:

$ProgressBar;<Title>;<Description>;<MaximumTimeInSeconds>

Arguments:

<Title>: Title of the progress bar dialog. Space characters are allowed. This argument is required.

<Description>: Description of the progress bar dialog. Space characters are allowed. This argument is required.

$PROMPTBYID¶

AutomationPanel supports this message, $PromptByID, primarily for BTS. However, it could be used by other applications as well. It is used to put up a multi-button prompt with corresponding actions where each instance of the prompt can have a specific ID. Therefore, the system can support multiple prompts with different IDs simultaneously as is required in the multi-circuit BTS system.

Version:
3.5

Syntax:

$PromptByID ID=1|B1=OK,Data_Engine>>SET flgAns 1,Exit|B2=Cancel,,Exit|B3=Do Nothing|T=Circuit Fault|BC=#663399|TC=0,0,0|L1=text|L2=Some more text

Arguments:

ID = ID of form to create
B1 = Button 1 label, Button 1 action, Exit on completion
B2 = Button 2 label, Button 2 action, Exit on completion
B2 = Button 3 label, Button 3 action, Exit on completion
T = Dialog Title
BC = Background Color
TC = Text Color
L1 = Label 1
L2 = Label 2

  • The Background Color and Text Color can be a numeric (663399) value or RGB (0,0,0).
  • This message will not create a new form if the ID already exists.
  • You can define a one, two, or three button dialog.
     

$PROMPTCLOSEBYID¶

Closes the prompt from the $PromptByID command.

Version:
3.5

Syntax:

PromptCloseByID <ID=#>

Arguments:

<ID=>: ID of form to close, such as ID=1
 

$PROMPTER;message;$INPUT;channelname1;channelname2;buttontext1;buttontext2¶

Displays an edit box with a right and left button. Clicking Button1 (left) sets the channelname2 channel to 1 and writes the input text to the channelname1 channel; clicking Button2 (right) sets the channelname1 channel to 0, and it does not write the input text to the channelname1 channel.

This mailslot message is only supported in AutomationPanel.

Version:
3.6

Syntax:

$PROMPTER;<message>;$INPUT;<virtualstringchannel>;<virtualoutputchannel>;<Button1Text>;<Button2Text>

 

$RESTORE¶

Restores the AutomationPanel interface to its original size and location. See $MINIMIZE.

Version:
3.6

Syntax:

$RESTORE

 

$SCHEDULEACTION¶

Controls the Schedule Viewer panel's toolbar buttons and combo box.

Syntax:

$ScheduleAction;<ScheduleName>,<Action>

Arguments:

<ScheduleName>: Name of the schedule file with or without the .sc2 extension.

<Action>: Options include: Start, Stop, Pause, Resume, NextStep, GotoStep, and Select.

NOTE: When action is set to "Select", it selects a schedule from the schedule combo box. If schedule name is "None" or "", it will unselect the current schedule (i.e., no schedule will be selected).

 

$SetItemProperty¶

Sets the value of a property for a panel/display object. This message requires a Property ID, which is specified in Property Manager.

Syntax:

$SetItemProperty;<PropertyID>=<Value>

Arguments:

<PropertyID>: The name of the Automation ID and property name. This is visible at the bottom of the Property Manager dialog when you select a property that supports automation.

<Value>: The value that the panel/display object property is set to.

Property ID Syntax:

<AutomationID>.<PropertyName>

Arguments:

<AutomationID>: The name assigned to the panel or object in Property Manager.

<PropertyName>: The internal name for the property. (This is not necessarily the name displayed in the left column of Property Manager.)

NOTE: If an object doesn't have an Automation ID defined, then none of its properties can be modified, and Property Manager will not display a PropertyID.

 

$SHOWGROUP¶

Shows panels with the specified Group ID.

NOTE: To show panels in layouts that were launched using DspDrv_AP.exe, the destination of the mailslot must be changed from "Softpanel" to the filename (without the file extension) of the layout. In other words, if the GroupID is shared across a combination of layouts, some launched using DspDrv_AP.exe and some open in AutomationPanel, you would have to send multiple $SHOWGROUP mailslots; one to Softpanel and one to each layout opened using DspDrv_AP.exe. The mailslot affects all panels with the specified Group ID in any layout that is currently open (e.g., docked layouts, tabs, floating tabs, layout toolbars).

Version:
4.1

Syntax:

$SHOWGROUP;<Group ID>

 

$HIDEGROUP¶

Hides panels with the specified Group ID.

NOTE: To hide panels in layouts that were launched using DspDrv_AP.exe, the destination of the mailslot must be changed from "Softpanel" to the filename (without file extension) of the layout. So, if the GroupID is shared across a combination of layouts, some launched using DspDrv_AP.exe and some open in AutomationPanel, you would have to send multiple $HIDEGROUP mailslots; one to Softpanel and one to each layout opened using DspDrv_AP.exe. The mailslot affects all panels with the specified Group ID in any layout that is currently open (e.g., docked layouts, tabs, floating tabs, layout toolbars).

Version:
4.1

Syntax:

$HIDEGROUP;<Group ID>

 

$SHOWTOOLBAR¶

Shows a toolbar hidden from the main AutomationPanel interface.

Syntax:

$SHOWTOOLBAR;<Display>

 

$SAVELAYOUT¶

Saves a layout in AutomationPanel.

Version:
4.1

Syntax:

$SAVELAYOUT;<LayoutName>;[ForceSave];[StatusChan]

Arguments:

<LayoutName>: The file name of the layout that you want to save. Do not include the file path or the file extension as part of this argument.

[ForceSave]: (Optional) Force saving changes to the layout without confirmation. Where: true - forced saving, false - show save confirmation dialog. If this argument is missing, the user will always be prompted to save changes.

[StatusChan]: (Optional) Name of the virtual output channel set to the returned value from the message. Where: 1 - busy, 0 - save successfully or cancel by confirmation dialog, -1 - error occurred.
 

$SysLogExport¶

Exports the values from the System Log file to a specified file.

Version:
4.4

Syntax:

$SysLogExport;<SysLogFile>;<Start Time>;<End Time>;<Output File>;<StatusChan>;<silent>

Arguments:

<SysLogFile>: The file path to the current active System Log file. Macros (e.g., $SYSTEMDIR) are supported for this argument.

<Start Time>: The timestamp (in YYYY/MM/DD HH:mm:ss.uuu format) to start collecting values at in the System Log file. If not specified, then values will be collected from the beginning of the System Log file.

<End Time>: The timestamp (in YYYY/MM/DD HH:mm:ss.uuu format) to stop collecting values at in the System Log file. If not specified, then values will stop being collected at the end of the System Log file.

<Output File>: The path to the outputted file where the collected values will be exported to. Macros (e.g., $SYSTEMDIR) are supported for this argument.

<StatusChan>: (Optional) The name of the virtual output channel used to display the status of the message, where:
1 = Busy
-1 = Invalid Time Range
-2 = Error reading the System Log File
-3 = Error opening Output File
0 = Operation was Successful

<Silent>: (Optional) Hides messages for errors or successes for the operation. In order to receive feedback in case of any errors, include the <StatusChan> argument in the message.

Example:

$SysLogExport;$DATADIR\PR-001\SystemLog.slg;2024/09/23 17:50:03.450;2024/09/23 18:14:38.570;$DATADIR\PR-001\OutputFile.csv;Cntr1;silent
