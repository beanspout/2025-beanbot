Overview¶

The mailslot address for DataEngine is Data_Engine. All messages should reference an address like this:

\\.\mailslot\Data_Engine>>Your Message
NOTE: Use the tilde ( ) to escape characters in mailslot messages.
WARNING: Do not include a semi-colon in any messages to Data_Engine. Any text following the semi-colon, including the semi-colon, will not be included as part of the message.

Channel Messages¶

Set¶

Sets a channel's value in the RDB to a given value.

Syntax:

SET <ChannelName> <NewValue>

 

Set AO¶

Sets an analog output channel's value in the RDB to a given value.

Syntax:

SET AO <ChannelName> <NewValue>

 

Set DO¶

Sets a digital output channel's value in the RDB to a given value.

Syntax:

SET DO <ChannelName> <NewValue>

 

Set EU Range¶

Sets an analog output channel's High EU and Low EU RDB field values to the given range.

Syntax:

Set EU Range <ChannelName> <LowEU> <HighEU>

 

Set Signal Range¶

Sets an analog output channel's High Rail and Low Rail RDB field values to the given range.

Syntax:

Set Signal Range <ChannelName> <LowRail> <HighRail>

 

Set VO¶

Sets a virtual output channel's value in the RDB to a given value.

Syntax:

SET VO <ChannelName> <NewValue>

 

Set VS¶

Sets a virtual string channel's value in the RDB to a given value.

Syntax:

SET VS <ChannelName> <NewValue>

 

Data Logging Messages¶

CheckLogHeader¶

Checks if a set of header rows will be added to the next data log.

Syntax:

CheckLogHeader <LogName> [Status]

Arguments:

<LogName>: The name of the log file to check.

[Status]: The virtual output channel used to display the result.

Status Values:

Value Description
1 Header rows will be added
0 Header rows won't be added
-1 Error parsing the log file or .lol file
-2 Unsupported data file type: .mdf or .bin (for transient logs only)

Example:

MESSAGE "\\.\mailslot\Data_Engine>>CheckLogHeader MyDataFile voResultChannel"

 

DataArchive¶

Creates or opens the specified directory as the project folder in the $DATADIR. Logged Blackbox data, the system log, and logged data files are saved in this directory. This mailslot modifies the DRHoldForReviewDir= powertek.ini setting.

Syntax:

DataArchive New|Open <FolderName>

Arguments:

New | Open: The first argument can either be "New" or "Open." This denotes whether iTest should create a new folder or open an existing folder.

FolderName: This argument follows the Windows naming convention for folders. The following invalid characters will be replaced with an underscore character: /:*?"|<>. Spaces in the data project folder name are supported.

Example:

MESSAGE "\\.\mailslot\Data_Engine>>DataArchive Open MyProject"

 

Log Cancel¶

Causes Data_Engine to cancel the log name listed. The existing data that has been gathered will be discarded.

Syntax:

LOG CANCEL <LogName>

 

Log Parse¶

Forces the logsetup.dat and all Log Order List (.lol) files to be read into memory. This message is used after changing, adding, or deleting data log configurations or Log Order Lists in Test Manager.

NOTE: Changes to Blackbox properties or Blackbox.lol requires an iTest restart. Log Parse will not update these during runtime.

Syntax:

LOG PARSE

 

Log Start¶

Starts a data log. Only one instance of a log can be started at a time.

Syntax:

LOG START <LogName> [Status]

Arguments:

LogName: The name of the data log to start.

[Status]: The virtual output channel that is used to display the status of the log.

Example:

MESSAGE "\\.\mailslot\Data_Engine>>LOG START NewLog1 voStatusChannel"

Status Values:

Value Description
1 Log Started
0 Idle
-1 Error: unable to start the log due to missing log order list (.lol) file
-2 Error: log already running
NOTE: If your solution has logic that relies on the value of the status channel to perform an action, then it is important to set the channel to an invalid value, such as -9999, before starting the log. This enables the code to synchronize with log starts and stops by waiting for the status to change to a valid status code.

 

Log Stop¶

Stops a single data log or all running data logs. A log file is generated that contains data that has been accumulated up to this point.

Syntax:

LOG STOP All|<LogName>

Arguments:

All | LogName: Stops the running data log. Use of the "All" string instructs Data_Engine to stop all running data logs.

Examples:

MESSAGE "\\.\mailslot\Data_Engine>>LOG STOP All" 
MESSAGE "\\.\mailslot\Data_Engine>>LOG STOP NewLog1"

 

StartLog¶

Starts a data log. Only one instance of a log can be started at a time. This mailslot message functions the same as Log Start.

Syntax:

STARTLOG <LogName> [Status]

Arguments:

LogName: The name of the data log to start.

[Status]: The virtual output channel that is used to display the status of the log.

Example:

MESSAGE "\\.\mailslot\Data_Engine>>STARTLOG NewLog1 voStatusChannel"

Status Values:

Value Description
1 Log Started
0 Idle
-1 Error: unable to start the log due to missing log order list (.lol) file
-2 Error: log already running
NOTE: If your solution has logic that relies on the value of the status channel to perform an action, then it is important to set the channel to an invalid value, such as -9999, before starting the log. This enables the code to synchronize with log starts and stops by waiting for the status to change to a valid status code.

 

StopLog¶

Stops a single data log or all running data logs. A log file is generated that contains data that has been accumulated up to this point. This mailslot message functions the same as Log Stop.

Syntax:

STOPLOG All|<LogName>

Arguments:

All | LogName: Stops the running data log. Use of the "All" string instructs Data_Engine to stop all running data logs.

Examples:

MESSAGE "\\.\mailslot\Data_Engine>>STOPLOG All" 
MESSAGE "\\.\mailslot\Data_Engine>>STOPLOG NewLog1"

 

UpdateMaxTransportDelay¶

Updates the voSysMaxTransportDelay system channel's value to the maximum time (in seconds) of any iTest channel that has a defined transport delay.

Syntax:

updatemaxtransportdelay

Driver Messages¶

Driver mailslot messages are communicated differently depending on whether the message is being sent to a distributed or non-distributed type system. For non-distributed systems, the message is sent directly to the driver's defined mailslot.

RTDriverLoad¶

The RTDriverLoad mailslot message enables you to dynamically start a driver on a distributed system (e.g., RTX_DIST or Xenomai). If the driver is currently running, it will be stopped and restarted. This message does support the use of the "NODE NAME" advanced driver setting from the .tdaddin component when working with INTime as the real-time operating system.

Syntax:

 RTDriverLoad <driver>

Arguments:

driver: The name of the driver's mailslot. Typically, it will match the name of the driver. This information is detailed in each driver's help documentation and, in some cases, can be configured by the driver's .tdaddin component.

Example:

 MESSAGE "\\.\mailslot\Data_Engine>>RTDriverLoad EtherCAT_1"

 

RTDriverMsg¶

The RTDriverMsg mailslot message enables you to send commands to drivers running on a distributed system (e.g., RTX_DIST or Xenomai). The RTDriverMsg message is first sent to Data_Engine, which then passes the included command to the indicated driver on the distributed system.

Syntax:

 RTDriverMsg <driver> <command>

Arguments:

driver: The name of the driver's mailslot. Typically, it will match the name of the driver. This information is detailed in each driver's help documentation and, in some cases, can be configured by the driver's .tdaddin component.

command: command to be sent to the driver's mailslot.

Example:

 MESSAGE "\\.\mailslot\Data_Engine>> RTDriverMsg EtherCAT_1 SetLogLevel:All"

 

Dynamic Calculation Messages¶

CalcEngine Start¶

Stops the current execution of CalcEngine and starts a new CalcEngine process. Alternatively, the mailslot can also designate the INtime node to start the CalcEngine process on.

Syntax:

CalcEngine Start [-nNodeX]

Arguments:

[-nNodeX]: Optional argument. This argument defines the INtime node to run the CalcEngine RTA process on. X is the node designation; this is typically A, B, or C (e.g., [-nNodeA] starts CalcEngine_rt4x.rta on node A).
 

CalcEngine Stop¶

Stops the current execution of CalcEngine.

Syntax:

CalcEngine Stop

 

Dynamic Name Change Messages¶

LoadAKAFile¶

Causes Data_Engine to load a new .aka file while iTest is running. Data_Engine reloads the customer names in the RDB and also rebuilds the internal customer name tables on both the PC and the real-time systems. This message is used to support dynamic customer names.

Syntax:

LOADAKAFILE <CustomerNameFile>

Arguments:

CustomerNameFile: The given filename must include the (.aka) file extension.
 

LoadAliasFile¶

Causes Data_Engine to load a new alias file while iTest is running. Data_Engine reloads the alias names in the RDB.

Syntax:

LoadAliasFile <AliasFileName>

Arguments:

AliasFileName: The given filename does not need to include the (.als) file extension.
 

INI File Messages¶

INIRead¶

Reads the value of a keyword=value pair in an .ini file and sets a channel's value in the RDB. If a return channel is not specified, then Data_Engine assumes that the name of the channel to be set is the same as the given keyword. Data_Engine uses the value from the first found instance of the given keyword in the .ini file.

Syntax:

INIRead <FileName> <Keyword> [ReturnChannel] [aryIDx=n]

Arguments:

FileName: The filename must include the .ini file extension. If a fully-qualified file path is not given, then the file path defaults to the $SUPPORTDIR.

Keyword: This argument is not case-sensitive. Use of a customer or module name in the message to find a channel name in the .ini file and vice versa is not supported.

[ReturnChannel]: Optional argument. The use of local variables, defined in VCL using VCLNUM or VCLSTR, for this argument is not supported.

[aryIDx=n]: Optional argument. If the keyword in the file is the name of an un-indexed array channel, then the value 'n' from this argument will be used as the array index (element) of the channel to be set.

Examples:

MESSAGE "\\.\mailslot\Data_Engine>>INIRead $SUPPORTDIR\master.ini voValue1 voValue2" 
* results in the voValue2 channel being set to the value associated with the voValue1 keyword in the master.ini file

MESSAGE "\\.\mailslot\Data_Engine>>INIRead $SUPPORTDIR\master.ini voaArray[1]" 
* results in voaArray[1]  being set to the value associated with the voaArray[1] keyword in the master.ini file

MESSAGE "\\.\mailslot\Data_Engine>>INIRead $SUPPORTDIR\master.ini voaArray" 
* results in voaArray[0] being set to the value associated with the voaArray keyword in the master.ini file

MESSAGE "\\.\mailslot\Data_Engine>>INIRead $SUPPORTDIR\master.ini voaArray aryIdx=2" 
* results in voaArray[2] being set to the value associated with the voaArray keyword in the master.ini file

 

INIReadFile¶

Reads the specified file as an .ini type file. .ini files use a keyword=value pair format. If the keyword on a line of the file matches the name of a channel in the RDB, then the channel is set to the value portion of the keyword=value pair.

Syntax:

INIReadFile <FileName> [aryIDx=n]

Arguments:

FileName: A fully-qualified path must be given, including the file extension.

[aryIDx=n]: Optional argument. If any keywords in the file matches the name of an unindexed array channel, then the value 'n' from this argument will be used as the array index (element) of the channel to be set.
 

INIReadMany¶

Reads the values for the given keywords from an .ini file and sets the matching iTest channels to these values. .ini files use a keyword=value pair format.

Syntax:

INIReadMany <FileName> <Keyword1> <Keyword2> ... <Keywordn> [aryIDx=n]

Arguments:

FileName: The filename must include the .ini file extension. If a fully-qualified file path is not given, then the file path defaults to the $SUPPORTDIR.

Keyword1...n: This argument is not case-sensitive. Use of a customer or module name in the message to find a channel name in the .ini file and vice versa is not supported.

[aryIDx=n]: Optional argument. If any keywords in the file matches the name of an unindexed array channel, then the value 'n' from this argument will be used as the array index (element) of the channel to be set.

Example:

MESSAGE "\\.\mailslot\Data_Engine>>INIReadMany $SUPPORTDIR\master.ini voNumeric1 voNumeric2 vsString1"

 

INIWrite¶

Updates the value of a keyword=value pair in an .ini file to either a given value or a given iTest channel's value. Data_Engine also sets the channel to the given value, if provided. All found instances of the keyword in the .ini file are updated. If a matching keyword is not found in the .ini file, then a new keyword=value pair is added to the end of the .ini file.

No type checking is performed, so it is possible to set a keyword, intended to store a numeric value, to a string value in the .ini file.

Syntax:

INIWrite <FileName> <Channel> [Value] [aryIDx=n]

Arguments:

FileName: The filename must include the .ini file extension. If a fully-qualified file path is not given, then the file path defaults to the $SUPPORTDIR.

Channel: Use of a customer or module name in the message to find a channel name in the .ini file and vice versa is not supported.

[Value]: Optional argument. If this argument is not provided, then Data_Engine writes the current value of the channel to the .ini file. If this argument is provided, then the channel's value in iTest will also be set to this argument.

[aryIDx=n]: Optional argument. If the Channel argument is the name of an unindexed array channel, the value 'n' from this argument is used as the array index (element) of the channel when setting the value.

Examples:

MESSAGE "\\.\mailslot\Data_Engine>>INIWrite $SUPPORTDIR\master.ini voValue1 1" 
* results in the voValue1 keyword's value in the master.ini file being set to 1 and the voValue1 channel's value being set to 1

MESSAGE "\\.\mailslot\Data_Engine>>INIWrite $SUPPORTDIR\master.ini voArray[1]" 
* results in the voArray[1] keyword's value in the master.ini file being set to the current value of voArray[1]

MESSAGE "\\.\mailslot\Data_Engine>>INIWrite $SUPPORTDIR\master.ini voArray aryIdx=1" 
* results in the voArray keyword's value in the master.ini file being set to the current value of voArray[1]

 

INIWriteFile¶

Causes Data_Engine to write the specified file as an .ini file. If the keyword on a line of the file is the name of a channel in the RDB, then the value of that keyword=value pair, in the file, is updated to the current value of that iTest channel. .ini files use a keyword=value pair format.

Syntax:

INIWriteFile <FileName> [aryIDx=n]

Arguments:

FileName: A fully-qualified path must be given, including the file extension.

[aryIDx=n]: Optional argument. If any keywords in the file matches the name of an unindexed array channel, then the value 'n' from this argument will be used as the array index (element) of the channel when reading and setting the channel's value.

NOTE: If the array channel is listed as an unindexed channel and listed again as an indexed array channel with the same element as the aryIDx argument value 'n', then both entries in the .ini file will be updated.

 

INIWriteMany¶

Updates the values of a set of keyword=value pairs in an .ini file to the given channels' values. All found instances of a keyword in the .ini file that match a given channel will be updated. If a given channel does not have a matching keyword in the .ini file, then a new keyword=value pair is added to the end of the .ini file. .ini files use a keyword=value pair format.

Syntax:

INIWriteMany <FileName> <Channel1> <Channel2> ... <Channeln> [aryIDx=n]

Arguments:

FileName: The filename must include the .ini file extension. If a fully-qualified file path is not given, then the file path defaults to the $SUPPORTDIR.

Channel1...n: This argument is not case-sensitive. Use of a customer or module name in the message to find a channel name in the .ini file and vice versa is not supported.

[aryIDx=n]: Optional argument. If any channels are the name of an unindexed array channel, then the value 'n' from this argument will be used as the array index (element) of the channel when reading and setting the channel's value in the RDB.

Example:

MESSAGE "\\.\mailslot\Data_Engine>>INIReadMany $SUPPORTDIR\master.ini voNumeric1 voNumeric2 vsString1"

 

Limits Messages¶

Limits Activate¶

This message either turns on the global limits flag or, if one or more channel names are given, sets the channel's Limits Active RDB field value to 1.

Syntax:

Limits Activate All|<Channel1...Channeln>

Arguments:

All | channel1...channeln: This argument is either the string "All" or a space-delimited list of iTest channels. If this argument is "All," the global limits flag is turned on. To turn on a specific channel's limits, use the channel's channel, customer, or module name as this argument.

Examples:

MESSAGE "\\.\mailslot\Data_Engine>>Limits Activate All" 
MESSAGE "\\.\mailslot\Data_Engine>>Limits Activate spdDt1Dyno aiDt1DynoTorque"

 

Limits Deactivate¶

This message either turns off the global limits flag or, if one or more channel names are given, sets the channel's Limits Active RDB field value to 0.

Syntax:

Limits Deactivate All|<Channel1...Channeln>

Arguments:

All | channel1...channeln: This argument is either the string "All" or a space-delimited list of iTest channels. If this argument is "All," the global limits flag is turned off. To turn off a specific channel's limits, use the channel's channel, customer, or module name.

Examples:

MESSAGE "\\.\mailslot\Data_Engine>>Limits Deactivate All" 
MESSAGE "\\.\mailslot\Data_Engine>>Limits Deactivate spdDt1Dyno aiDt1DynoTorque"

 

Limits Group Apply¶

Applies a Limit Group. If LimitsVer=3.2 in the powertek.ini file, then the applied limit group replaces the current limit settings. Otherwise, the limit group is applied with a layered approach.

Syntax:

Limits Group Apply <group>

Arguments:

group: Is the name of the Limits Group to be applied. Do not include the .lim file extension.

NOTE: If LimitsVer=3.2, this mailslot message will reactivate limits on channels that had been individually deactivated using either the Limits Deactivate mailslot message to Data_Engine or the Limits Deactivate VCL command.

 

Limits Reactivate¶

Activates all channels that have been deactivated individually. To be activated, a channel must have limits defined and its Limits Active RDB field value must be 0. If this criteria is met, the channel's Limits Active RDB field value is set to 1.

Syntax:

Limits Reactivate

 

Miscellaneous Messages¶

BackupLogFiles¶

Copies the current iTest log files from both the GUI and RT PC to the $SYSTEMDIR/Log/Archives folder.

CopyField¶

Copies RDB field value(s) from one iTest channel to another iTest channel or internal variable. If an RDB Field argument is not provided as part of the SOURCE argument, then the current value of the <SourceChannel> is used. If an RDB Field argument is not provided as part of the <DestinationChannel> argument, then the <DestinationChannel> is set to the value passed from the SOURCE argument if it not a string channel. When writing to a string channel (virtual string or virtual string array), the String Value RDB Field argument must be included as part of the DEST argument.

If the destination channel is not in the database, the program will assume that the destination is an internal variable and send the Set VO or Set VS commands to Data_Engine to set the internal variable's value. If sequential fields are used (e.g., calibration coefficients C0-C6), then brackets may be used to indicate that you want to access a specific element within that array. This element number would most likely be an intervar or database channel used for looping over all the index values.

This function returns a 1 upon success.

Syntax:

CopyField;SOURCE=<SourceChannel>|<Field>[<ElementNumber>];DEST=<DestinationChannel>|<Field>;RETURN=<ReturnChannel>

Required Arguments:
SOURCE=<SourceChannel>[ | <Field>[ [<ElementNumber>] ] ]
DEST=<DestinationChannel>[ | <Field>]

Optional Arguments:
RETURN=<ReturnChannel>

Examples:

MESSAGE "\\.\mailslot\Data_Engine>>CopyField;SOURCE=ChannelIn|SpanEU;DEST=ChannelOut|High Warning;RETURN=CalState" 
* sets ChannelOut's High Warning RDB field value to ChannelIn's SpanEU RDB field value

MESSAGE "\\.\mailslot\Data_Engine>>CopyField;SOURCE=ChannelIn;DEST=ChannelOut|High EU;RETURN=CalState" 
* sets ChannelOut's High EU RDB field value to ChannelIn's current value

MESSAGE "\\.\mailslot\Data_Engine>>CopyField;SOURCE=ChannelIn;DEST=ChannelOut;RETURN=CalState" 
* sets ChannelOut's value to ChannelIn's current value

MESSAGE "\\.\mailslot\Data_Engine>>CopyField;SOURCE=ChannelIn|Alias;DEST=vsChannel|String Value;RETURN=CalState" 
* sets vsChannel's value to ChannelIn's Alias RDB field value

MESSAGE "\\.\mailslot\Data_Engine>>CopyField;SOURCE=Channel1|C[%<i>%];DEST=Channel2[%<i>%]" 
* (where i is an intervar used for looping over values in the coefficient array)

 

NOTE: CopyField is an exception to the rule of semi-colon use in mailslot messages to Data_Engine.

 

Die¶

Sends the Shutdown command to vxin_exec, closes the TCP/IP socket, and then closes itself.

Syntax:

DIE

 

GenericMsg¶

This is a general utility mailslot message used to send commands directly to the vxin_exec. Use of this mailslot message is detailed with any feature that takes advantage of it.

Syntax:

 
GenericMsg <command> <argument1>...<argumentn>

Examples:

MESSAGE "\\.\mailslot\Data_Engine>>GenericMsg ShowChannel spdDt1Dyno" 

* displays the real-time value of the spdDt1Dyno channel in the vxin_exec command window


MESSAGE "\\.\mailslot\Data_Engine>>GenericMsg MasterSchedule NotifyEvents <Text>" 

* Adds the <Text> to the Master Schedule Event Log with the current step name; if there isn't a current step, then it writes "???".
* In this example, <Text> is a placeholder for any string no longer than 255 bytes; longer strings are truncated to 255 bytes.
* This particular GenericMsg only works if the following setting is in the powertek.ini file: MasterScheduleLogEvents=TRUE.

 

Load Tables.cfg¶

Updates the Lookup tables with the current contents of the tables.cfg file during runtime.

Syntax:

load tables.cfg

 

ResetCircleBuffer¶

Resets buffered data in CircularBuffer.

Syntax:

ResetCircleBuffer <ChannelName>

Arguments:

channelname: virtual output channel used to store the result of the mailslot message.

  • -1 = Data_Engine did not create a circle buffer
  • 0 = Invalid circle buffer instance
  • 1 = success

 

SysLog Event Add¶

Adds a line to the System Log stored in the SystemLog.slg file.

Syntax:

SysLog Event Add <Type> <Text>

Arguments:

type: This argument populates the Type column in the System Log and must not contain any white-space characters.

text: This argument populates the Description column in the System Log.
 

PID Control Loop Messages¶

PID Group Apply¶

Causes Data_Engine to read the specified PID Group from the $SUPPORTDIR\PID\pidgroups.cfg file, apply the found coefficients (P, I ,D) to the PIDs in the group, sends the PID Modify mailslot to Data_Engine, and logs an entry in the data_engine.log file. If the pidgroups.cfg file does not exist, then no action is performed.

The PIDTune utility maintains the pidgroups.cfg file via its File > Save Settings to Group menu options.

Syntax:

PID Group Apply <group>

Arguments:

group: PID Group to apply.

Example:

Given the following contents in the pidgroups.cfg file:

$GROUP HS1
SPEED.HS1    P=154.411    I=0.005    D=0.00509
$END-GROUP

The following VCL will result in the Speed.HS1 PIDs being modified:

+0.01 MESSAGE "\\.\mailslot\Data_Engine>>PID Group Apply HS1"

 

PID Initialize¶

If a PID file is specified for the LastModeHS1, LastModeHS2, LastModeLSx, and LastMode<pidgroup> fields in the powertek.ini file, then Data_Engine starts the PID using the file specified. The PID is not started if the listed PID file is not defined in the solution.

Syntax:

PID Initialize

Example:
Given the powertek.ini settings are as follows:

LastModeHS1=Speed.HS1
LastModeHS2=Position.HS2
LastModeDyno1=CoolantTemp.Dyno1

The following VCL will start the Speed.HS1 loop for the HS1 group, Position.HS2 loop for the HS2 group, and CoolantTemp.Dyno1 for the Dyno1 loop.
+0.01 MESSAGE "\\.\mailslot\Data_Engine>>PID Initialize"

 

PID Modify¶

Changes the specified PID to the mode specified in the PID file.

Syntax:

PID Modify <PIDgroup> <PIDfile>

Example:

+0.01 MESSAGE "\\.\mailslot\Data_Engine>>PID Modify HS1 Speed.HS1"

 

PID Start¶

Starts a PID. Data_Engine looks at the extension of the specified PID file to determine which PID group to start. Data_Engine then starts this PID group with the given file.

Syntax:

PID Start <PIDfile>

Example:

+0.01 MESSAGE "\\.\mailslot\Data_Engine>>PID Start Speed.HS1"

 

PID Stop¶

Stops the specified PID.

Syntax:

PID Stop <PIDgroup>

Example:

+0.01 MESSAGE "\\.\mailslot\Data_Engine>>PID Stop HS1"

 

Procedure Messages¶

Sequence Abort¶

Aborts a running procedure. This mailslot message will not abort a child procedure that was started using the EXECUTE VCL command. In this case, you must abort the parent procedure to abort the child procedure.

Syntax:

SEQUENCE ABORT <ProcedureName>

 

Sequence Resume¶

Resumes a suspended procedure.

Syntax:

SEQUENCE RESUME <ProcedureName>

 

Sequence Start¶

Starts another procedure that executes in parallel with the parent procedure. After this mailslot message is sent, the parent procedure will continue on to the next instruction without waiting for the child procedure to complete. By default, a new instance of the child procedure will be started even if one is already running unless AllowMultipleProcInstances=FALSE is present in the powertek.ini file and the child procedure does not accept arguments.

Syntax:

SEQUENCE START <ProcedureName>

Additionally, you can add the VCL SET command to the SEQUENCE START mailslot message to capture the returned value of a called procedure.

Syntax:

SEQUENCE START SET <ChannelName> <ProcedureName>

 

Sequence Suspend¶

Suspends a running procedure until the SEQUENCE RESUME mailslot message is sent.

Syntax:

SEQUENCE SUSPEND <ProcedureName>

 

Statistics Messages¶

Statistics Activate¶

Sends a packet to vxin_exec to activate statistics measurements for all iTest channels or for a given group of iTest Channels. A channel must have the Statistics Available RDB field value equal to 1 to be activated.

Syntax:

Statistics Activate [<Channel1...Channeln>]

Arguments:

[channel1...channeln]: Optional argument. This argument is a space-delimited list of iTest channel, customer, or module names.
 

Statistics Deactivate¶

Sends a packet to vxin_exec to deactivate statistics measurements for all iTest channels or for a given group of iTest Channels.

Syntax:

Statistics Deactivate [<Channel1...Channeln>]

Arguments:

[channel1...channeln]: Optional argument. This argument is a space-delimited list of iTest channel, customer, or module names.
 

Statistics Reset¶

Resets statistics measurements for all iTest channels or a given group of iTest Channels for which the Statistics Available RDB field is equal to 1. This deletes the current buffer contents for each channel and can, optionally, resize each channel's statistics measurement window.

If the optional [windowsize] argument(s) are not included, then all measurement windows are resized to 10 seconds, by default.

Syntax:

Statistics Reset [WindowSize]|[<Channel1...Channeln>]|[<WindowSize1> <Channel1>...<WindowSizen> <Channeln>]

Arguments:

[windowsize]: Optional argument. The window size is in seconds. The maximum supported window size is 10 seconds.

[channel1...channeln]: Optional argument. This argument's format is a space-delimited list of iTest channel, customer, or module names.

[windowsize1 Channel1...windowsizen channeln]: Optional argument. This format is a combination of the two other formats with the window size followed by the affected channel's name.

Example:

MESSAGE "\\.\mailslot\Data_Engine>>Statistics Reset 10" 
* results in all channels with Statistics Available equal to 1 having their statistics measurement windows...
* ...resized to 10 seconds and their statistics buffer contents deleted.

MESSAGE "\\.\mailslot\Data_Engine>>Statistics Reset voStat01 voStat02" 
* results in the voStat01 and voStat02 statistics buffer contents being deleted.

MESSAGE "\\.\mailslot\Data_Engine>>Statistics Reset 5 voStat01 10 voStat02" 
* results in the voStat01 and voStat02 statistics buffer contents being deleted. Also voStat01's statistics...
* ...measurement window would be set to 5 seconds and voStat02's statistics measurement window would be set to 10 seconds.

 

Step Conditions (EOS) Messages¶

Load EOS¶

Dynamically reloads the Step Condition files. Step Condition files are also referred to as End of State (EOS) files.

Syntax:

Load EOS

 

Update EOS¶

Data_Engine sends a packet to vxin_exec to update a given Step Condition with a new set of conditions. Step Conditions are also referred to as End of State (EOS) Conditions.

Syntax:

Update EOS <eosName> <eosCondition>

Arguments:

eosname: The name of the Step Condition to be updated.

eoscondition: The replacement EOS condition.
