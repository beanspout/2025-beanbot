BTSI Lab Systems Integration : Jenkins Good Practices & File Management

Created by Constantin Balan, last updated on Jan 04, 2023

Jenkins Logs

System Logs:

The Jenkins system logs are found in the <JENKINS_HOME>/logs/ folder on Windows systems. This is where you can find anything related to the running of the Jenkins process. If you ever have issues with Jenkins running/starting, you can check here for any errors being thrown.

Job Logs:

Each Jenkins job has the option to discard old builds. However if you are logged in as an admin you can navigate to:

Dashboard > Manage Jenkins > Configure System > Global Build Discarders

Here you can set a log rotation up for the jobs that will keep logs based max number of jobs, or based on number of days since the build ran. It is suggested to have a log rotation set up because otherwise the computer where the builds are running will eventually run out of space due to this mechanic being the only way to automatically clear logs on a Windows OS.

Jenkins Jobs

Jobs Directory:

The jobs directory contains directories of every single job you have set up on the Jenkins instance. Each job will have a config.xml file, as well as a sub-directory of builds. The sub-directory will have all of the build logs of the retained builds as well as a copy of the config file that was present at runtime. Workspaces:

Workspaces are the directories where Jenkins jobs are executed. Workspaces should be ephemeral (not permanent, short lived). So when finishing a job in a workspace, the best practice would be to configure the workspace to clear once it's complete. If there are and files left in the workspace when a new build runs, they could potentially affect the job in unforeseen ways. This is why making sure the workspace always is consistent when running is the most effective and reduces the chance of unnecessary build errors. Doing a workspace clear both before and after a build runs would be the best way of doing this. The clear after a build runs should only be done on successful builds because doing it on a failed build will potentially delete some evidence on why the build failed (We don't want that, that ain't good).
