Overview¶

Integrating your test systems into the LabCentral environment provides additional functionality that enables you to better reuse software. LabCentral is a server-based lab management tool that centralizes test cell management activity and data. From its different web-pages, you can manage test systems, monitor data, and create reports to obtain and view results for an entire test field rather than single systems all from a centralized location.

This document describes the ways in which LabCentral and iTest features overlap and the configurations necessary to successfully integrate the two environments.

NOTE: iTest can be used to store solutions in a LabCentral folder on your local machine. This is advantageous when (for example) you are working remotely yet still want to package your solutions in a way that it can be easily moved into LabCentral once you have access to a web connection.

 

LabAgent¶

LabCentral interacts with iTest by using LabAgent. LabAgent enables users to monitor system activity via a dashboard in LabCentral. During the iTest installation, LabAgent is automatically installed, and the SystemConfig dialog can be used to enable/disable LabAgent and modify the LabAgent port number. By default, LabAgent is enabled and uses port 80; the port number can be modified if port 80 is blocked or is in use by another process. When LabAgent is not connected, communication with LabCentral is lost. For more information, refer to iTest's SystemConfig and LabAgent's Installing and Configuring LabAgent documentation.

SystemConfig - LabAgent Settings

In addition to enabling the communication between iTest and LabCentral, LabAgent is used to maintain utilization prompt information received from the operator. LabCentral relies on this information to record and classify test system utilization states. The flag used to indicate the running state of your test system is constantly monitored. When it transitions from TRUE to FALSE (or vice versa), LabAgent waits for a defined number of minutes (this is specified in LabCentral) before displaying the Select a Status Change dialog. This dialog asks the operator to select the running or not-running reason for the state change. This feature enables LabCentral to properly classify the ways in which test systems are used during active testing and ways they are not used during an inactive state. For more information, refer to the LabCentral User Guide and LabAgent User Guide documentation.

You can modify the names of these channels by connecting to the LabAgent home page and selecting the Map Default Channels option in the navigation tree. The Default Channel Name Map page displays, allowing you to map the channel names to something more applicable to your solution. You can also change these channel names from LabCentral using the Master Names List feature.

Security¶

LabCentral can be used to set up security settings for each connected iTest system. This generally includes the following information:

  • Usernames and passwords
  • User groups and permissions

For more information about configuring security in LabCentral, refer to the LabCentral's Administering System Security documentation. The security settings can be overwritten locally and managed via the Security Editor; refer to the Managing Security documentation for more information. If you are connected to LabCentral, your local changes will be lost the next time an update is pushed down from LabCentral.

Synchronizing Files with LabCentral¶

Test Manager allows you to share files with LabCentral and distribute them globally to all iTest systems. When LabAgent is connected to LabCentral, you can synchronize, unsynchronized, and add new files to LabCentral. The following files/folders can be synced:

  • Battery Setup: Batteries
  • Calculations: 1Hz, 10Hz, 100Hz, System Rate
  • Test Development: Schedules
  • Data Logging: Log Header Forms
  • Presentation/Reports: Report Pages, Report Parameters
  • Picklists
NOTE: To use this feature, the IncludeLabScheduler config.ini setting must be set to FALSE.

When you synchronize files, an icon displays next to the file name. This icon is a visual indicator that communicates whether files have been synced. Green circles with a checkmark indicate that the file/folder is synced, red circles with an exclamation mark indicate that a synced file/folder has been modified and needs to be re-synced, and blue circles with a question mark indicate that the file does not exist on the server and needs to be added to LabCentral.

Icon Types

 

Synchronizing and Unsynchronizing Folders¶

The following process synchronizes an iTest folder to the LabCentral file repository. All changes made locally are overwritten the next time you start iTest unless you commit the changes back to LabCentral before exiting Test Manager. To synchronize/unsynchronize folders, do the following:

  1. Right-click the folder and select the Synchronize with LabCentral... option. The Synchronize with LabCentral dialog displays, prompting you to select a folder on the LabCentral server.
     
    Synchronize with LabCentral Dialog

     
  2. Specify the remote folder and files to synchronize using the drop-down lists. You can select the option to sync all files, which ensures that all files from the specified LabCentral location are copied to the local test system when iTest is initialized. You can also choose to only sync files that already exist locally. This option does not copy new files down; it only replaces files that are already exist on the local iTest system. When a folder is synchronized with LabCentral, a green checkmark icon is added to the folder and the files that are synced.
     
    Synchronization Type

     
  3. If you create a new file in Test Manager, or have existing local files, you will need to add the file(s) to LabCentral to enable synchronization. Once added to LabCentral, the file(s) will be available for other users. To add a file, right-click on the file and select the Add to LabCentral... option. The Add to LabCentral... dialog displays. Enter any applicable comments and click OK.
     
    Add to LabCentral... Dialog

     
  4. If you have made changes to a synchronized file, the green checkmark on the folder and file will change to a red exclamation mark. Before exiting Test Manager, you must commit these changes or they will be overwritten upon re-initialization. To commit changes, right-click the file with the red exclamation mark and select the Commit Changes to LabCentral... option. The Commit to LabCentral... dialog displays. Enter any applicable comments and click OK.
     
    Commit to LabCentral... Dialog

     
  5. To unsynchronize a folder, right-click the folder name and select the Unsynchronize option.

Importing and Exporting Files¶

One of the primary functions of LabCentral is to behave as a solution and module repository, enabling you to transfer files between LabCentral and iTest via uploads and downloads, where:

  • Solutions are combinations of module types, including devices, specimens, tests, etc. A solution is structured as a standard support folder with additional structures to enable the import and export of modules.
  • Modules are partial support folders that contain only the structures necessary for the model and the structures to connect a module to a solution.

There are several locations in Test Manager and SolutionBuilder that enable you to interact with LabCentral. If iTest is connected to LabCentral, these features will be available. The main purpose of these features is to allow files to be transferred from LabCentral down to iTest and vice-versa. This allows you to set up a centralized file management system in your lab.

There is an option to utilize the LabCentral Subversion Repository for controlling the import and export of modules. This feature automatically manages changes through revision history and user added comments. To use this feature, you must have the ModuleRevisionControl Config.ini setting set to TRUE.

For more information about importing and exporting modules, refer to the Modules in iTest documentation.

ModuleUploadFolder¶

The ModuleUploadFolder config.ini setting prevents developers from inadvertently deleting solutions that are intended to be more closely managed on LabCentral. With this setting, you can designate an incoming folder for all exported solutions and modules, allowing you to segregate the upload folder from the download folder. All modules and solutions uploaded to LabCentral from Test Manager or SolutionBuilder are saved to the folder specified in the ModuleUploadFolder setting. This should be a single name and is assumed to be a subfolder of the iCentralConfigFolder setting.

Example:

iCentralConfigFolder=\\LabCentral\Config\
ModuleUploadFolder=Incoming

This would cause all modules and solutions to be placed into the \\LabCentral\Config\Incoming folder.

Importing Solutions¶

You can copy a solution that is stored on LabCentral to your local system using Test Manager or SolutionBuilder. In Test Manager, select File > New > Solution or right-click on the solution name in the tree bar and select New. In SolutionBuilder, click the Create new solution... link on the homepage. The New Solution dialog displays.

New Solution Dialog

Select an existing solution to use as the base for the new solution. Enter the name of the new solution in the text field, and then click Create. The existing solution can be from LabCentral or your local iTest folder.

Exporting Solutions¶

From Test Manager or SolutionBuilder, you can export solutions to LabCentral or to your local drive. This allows your developers to share their solutions. Without a connection to LabCentral, exports are sent to the $SUPPORTDIR\Solutions folder when attempting to export to LabCentral.

There are two ways to export the entire solution while in Test Manager.

  1. Select File > Export to LabCentral > Solution from the main menu.
  2. Right-click the solution name in the navigation menu and click the Export option.

To export a solution using SolutionBuilder, click the Export icon from the bottom menu.

Export Icon

The Export Solution to LabCentral dialog displays. You can then export your solution to the folder specified in the config.ini file as iCentralConfigFolder=. This can be a LabCentral server or any network share.

Export Solution to LabCentral Dialog (example from Test Manager)

From this dialog, you can provide an alternative name for the destination solution as well as export the solution to your local drive or the LabCentral solution folder. If the solution name already exists on LabCentral and you are not using revision control, you will be prompted whether to overwrite the pre-existing solution. In addition, you can delete the local solution after the export is complete by checking the Delete local copy checkbox; this option is only available when exporting solutions from Test Manager while iTest is not running.

LabCentral Subversion¶

The LabCentral Subversion Repository enables you to import and export modules between Test Manager and Subversion. Revision control is automatically managed, so you can keep track of changes through revision history and user-added comments. To use this feature, the ModuleRevisionControl config.ini setting must be set to TRUE.

This setting determines where modules are uploaded and downloaded on LabCentral. If this setting is set to FALSE, then the normal LabCentral file share is used; the module is stored in the Modules\Device folder on LabCentral's Solutions And Modules page. If this setting is set to TRUE, then the LabCentral Subversion Repository is used; the modules are stored in the Modules folder on LabCentral's Manage Lab Files page.

FlexEdit¶

There are some areas of the FlexEdit interface that can utilize LabCentral services. If your system includes a FlexEdit interface for the operator to prepare tests with or perform other routine editing tasks, refer to the following sections. For more information about FlexEdit, refer to the FlexEdit documentation.

File Transfer for a <task>¶

If your FlexEdit configuration file has the <icentral> tag within a <task> element, and it points to a valid folder in the LabCentral Lab Files repository, it will add the Server Download and Server Upload buttons to the toolbar. The Server Download button allows the file to be downloaded from the specified LabCentral folder; the Server Upload button allows the file to be saved back to LabCentral with an operator specified name. For example, if a filename is in the tree called EngineInfo.ini and the <icentral>EngineData</icentral> tag appears in the XML file, then FlexEdit will look for a valid connection to LabCentral. If the EngineData folder contained two files called Model1234.ini and Model5678.ini, then the operator could select one of these files and overwrite the local file – EngineInfo.ini. If the operator changes the filename to Model8901.ini, they can then save it back to LabCentral using that filename.

Saving and Restoring Configurations¶

FlexEdit includes the Save Configuration... and Restore Configuration... menu options for uploading and downloading full configurations to/from LabCentral. The File > Save Configuration... option creates a ZIP file of the entire FlexEdit configuration and all relevant files and uploads it to LabCentral or a server share. The File > Restore Configuration option downloads the uploaded ZIP files and extracts them to the local system. The ZIP file is named by the operator and enables a test configuration to be transferred from one location in the lab to another. For more information, refer to the FlexEdit Design documentation.

Part Tracker¶

Part Tracker is a LabCentral tool that enables you to track usage information for different parts under test, such as engines, transmissions, treatment systems, or batteries. Tracked data is stored in a Part Tracker database which then can be viewed in LabCentral as a report or exported to different systems. For more information, refer to LabCentral's Using Part Tracker documentation.

The following sections detail how to use VCL to add new values or accumulators to a Part Tracker database. iTest utilizes the iCentralRead and iCentralWrite command line tools to send and retrieve parameters from the Part Tracker database.

iCentralRead¶

iCentralRead is used to read information from LabCentral to iTest.

Command-Line Syntax¶

Read Part Tracker information back from LabCentral to iTest:

iCentralRead /db <database> /prt <part id> /param <param-id> /statuschannel <statuschannel> /valtype <valtype>

Read every channel stored in the database for the part ID:

iCentralRead /db < database> /prt <part-id> /statuschannel <status-channel>/valtype <valtype>

Read a part tracker parameter to a specific iTest channel:

iCentralRead /db <database> /prt <part-id> [/param <param-id> /channel <channelname>] /valtype <valtype> /statuschannel <status-channel

Specify parameter names to map to channel names:

iCentralRead /db <database> /prt <part-id> /f <filename> /valtype <valtype> /statuschannel <status-channel>

For more information, including command-line switch definitions and status codes for iCentralRead, please refer to LabCentral's Interfaces to Part Tracker document.

The following table describes the switch definitions used above:

Read Part Tracker Data to iTest - Switch Definitions

Switch Description
[/aryidx <index>] An optional numeric array index to be appended to array type RDB channels read from the file specified by the /f switch.
/channel <channelname> The RDB channel that the read value is written to. This switch can only be used when /param is defined.
/db <database> The name of the part tracker database in LabCentral.
/f <filename> The name of the file to which all values are written. The file format is parameter=value where “parameter” represents the parameter name. Alternatively, you can use a standalone iTest channel to obtain a value from the RDB in iTest. Note: This can only be used when both /param and /channel are not used.
/param <param-id> The parameter name (also the default channel name) being read from the database.
/prt <part-id> The serial number or ID of the part (e.g., engine, battery, transmission).
/statuschannel <statuschannel> The iTest channel that iCentralRead.exe uses to store its error status when performing an operation. Negative numbers indicate error and positive numbers indicate success. The log
iCentralReadLog.txt will contain error messages. (For more information, see the Interfaces to Part Tracker document's iCentralRead Status Codes section.)
/valtype <valtype> Value type to store. Allowed values include total and lastvalue.

 

iCentralWrite¶

NOTE: When writing part tracker data to LabCentral, the system used for development must have LabAgent installed and connected as a test cell to LabCentral.

Use iCentralWrite to specify information to write to the LabCentral server (e.g., database name, part ID, parameter name, and parameter value). The tool performs the following:

  • Reads the $SYSTEMDIR setting in config.ini.
  • Creates a folder in $SYSTEMDIR called Transaction.
  • Creates an XML "transaction" file that contains the iCentralWrite request and a trigger.txt file.
  • Returns the iCentralWrite status.

Before using this feature, ensure you have all necessary components to begin. iCentralWrite requires the following:

  • iView.exe must be installed and running from the same folder as the iCentralAPI.dll and the C# application.
  • The ini.dll file must be in the same folder as the iCentralAPI.dll and the C# application. This file is installed with the iTest and LabAgent installs.
  • Config.ini must be located in the parent directory ($SYSTEMDIR) of the folder containing iCentralAPI.dll and the C# application (e.g., C:\ADT\iTest3.8).

Command-Line Syntax¶

iCentralWrite supports a single parameter syntax and a multi-parameter syntax.

Write a single parameter value:

iCentralWrite /db <database> /prt <part-id> /param <param-id> /val <value> /cmd <cmd> /statuschannel <statuschannel> [/createdb] [/dbdeffile <dbdeffile>] [/prtdeffile <prtdeffile>] [/deftype <deftype>] [/post [/trigger]]

Write multiple parameter values using data from a file:

iCentralWrite /db <database> /prt <part-id> /f <filename> /cmd <cmd> /statuschannel <statuschannel> [/createdb] [/channelnames] [/aryidx <index>] [/dbdeffile <dbdeffile>] [/prtdeffile <prtdeffile>] [/deftype <deftype>] [/post [/trigger]]

Part Tracker Input File¶

The /f command-line switch references an input text file. This input file maps Part Tracker parameters to channels and values, so you can set multiple parameters with one command.

The rows of an input file support 3 different formats which are specified below. The same file may utilize multiple row formats.

Input File Row Specification

Row Format Description
<Parameter> The channel name matching the Part Tracker parameter will have its value written to the parameter.
<Parameter>=<Value> The specified value will be written to the Part Tracker parameter.
<Parameter>=<Channel> The specified channel's value will be written to the Part Tracker parameter.

Example File Contents:

Voltage=5
SerialNumber=vsSerialNumber
UserID=vsUserId
voShift

Command-Line Switch Descriptions¶

The following table describes the switch definitions used above:

Write Part Tracker Data to LabCentral - Switch Definitions

Switch Description
[/aryidx <index>] An optional numeric array index to be appended to un-indexed array type channels in files specified by the /f switch.
/channelnames Used only with the /f switch. The switch that verifies if the "value" reference in a "parameter=value" pair is an existing channel and sets/increments the parameter's value by that channel's value from the RDB in iTest. This is an optional argument.
/cmd [SET | INCREMENT] The command used to determine how values are added to existing values in the database. The SET command sets the current value to the one you specify. The INCREMENT command adds the value you specify to the previous value in the database.
Example:
iCentralWrite.exe /db engine1 /prt 471AE7202 /param standid /val 2009 /cmd SET
OR
iCentralWrite.exe /db engine1 /prt 471AE7202 /param standid /val 2009 /cmd INCREMENT
/createdb The switch used to create the part tracker database in LabCentral if it does not already exist.
/db <database> The name of the part tracker database used to store tracked part data.
If the database does not exist and /createdb was not used, an Inbox notification will be sent to members of the Administrator group describing the error.
/dbdeffile <dbdeffile> A lab file path to the database’s default definition file relative to the database’s part template folder (see /deftype). This path uses forward slashes (e.g., qaBatteries/Batter1A.sc2).
/deftype [SC2 | INI | MSS] The type of definition files used by the database and its parts. This switch also determines the location of the database’s part template folder.
 
- SC2: (Default) .sc2 files from the PartTemplates folder will be used.
- INI: .ini files from the PartTemplatesINI folder will be used.
- MSS: .mss files from the PartTemplatesMSS folder will be used.
 
If the database already exists, its previous /deftype will remain.
/f <filename> The name of the file from which all values are written. The file format is parameter=value, where “parameter” represents the parameter name. Alternatively, you can use a standalone iTest channel to obtain a value from the RDB in iTest. This is used instead of /param <param-id>.
/param <param-id> The name for the part ID's parameter value being written to Part Tracker.
/post Sends the transaction XML file directly to LabCentral, skipping over LabAgent's local transaction queue.
/prt <part-id> The serial number or ID of the part (e.g., engine, battery, transmission). Part IDs are truncated at 64 characters.
/prtdeffile <prtdeffile> A lab file path to the part’s definition file relative to the database’s part template folder (see /deftype).
/statuschannel <statuschannel> The iTest channel that iCentralWrite.exe uses to store its error status when performing an operation. For more information, see the iCentralWriteStatus Codes section below.
/trigger Forces LabCentral to process all queued Part Tracker transactions upon receiving the command. This switch is only usable in tandem with the /post switch.
 
This switch requires LabCentral 3.5 or higher.
/val <value> The value to either set or increment the parameter.
NOTE: The following character types are allowed in the /db, /prt, and /param switches: alphanumeric, dash, period, @, and underscore. Any other characters will automatically be removed.

Create Part Tracker Database Remotely¶

Part tracker databases can be created remotely from iTest using command lines. The command-line syntax is represented by the following use cases:

To create a part tracker database for a file name with a single parameter:

iCentralWrite /db <database> /prt <part-id> /param <param-id> /val <value> /cmd <cmd> /statuschannel <statuschannel> /createdb

To create a part tracker database for a file name with many parameters:

iCentralWrite /db <database> /prt <part-id> /f <filename> /cmd <cmd> /statuschannel <statuschannel> /createdb

Specify Part Tracker Definition File Remotely¶

Part tracker definition files can be specified remotely from iTest for databases and part IDs. This process utilizes two different command lines; the syntax is represented below:

To specify a part tracker definition file for a database:

iCentralWrite /db <database> /prt <part-id> /param <param-id> /val <value> /cmd <cmd> 
/ statuschannel <statuschannel> /createdb /dbdeffile <database-definition-file> /deftype <SC2 | INI | MSS>

To specify a part tracker definition file for a part ID:

iCentralWrite /db <database> /prt <part-id> /param <param-id> /val <value> /cmd <cmd> 
/statuschannel <statuschannel>  /createdb /prtdeffile <part-definition-file>

Paths are relative to either the PartTemplates, PartTemplatesINI, or PartTemplatesMSS folder depending on the /deftype switch; if a subfolder is not specified, then the file paths will point to the root of the specified template folder.

Automatic Headers¶

In LabCentral, you can configure a list of channel names that are used when you search for data files in your lab. These channel names are automatically added to each iTest data file that is created on the iTest system. iTest creates the header of the DAT files using the names of channels listed in the designated UVL file, and then adds the list of channels that were configured on LabCentral. These headers can be defined using the LabCentral Master Names List page.

Master Names List Interface

LabCentral uses the information defined on this screen to create the iCentralParams.xml file which is downloaded to each iTest system and stored in the Execute folder. This file contains a section that is used by datareduction to add lab-wide searchable channel names to every data file header created. On your system, the data file header will be a super set of the channels in this file and the UVL file specified for your data log.

For example, if the iCentralParams.xml file on your system had a section that looked like this file: 

<SearchableHeaders>

<Header><Name>vsSysSpec1EngType</Name>
    <Label>Engine Type</Label>
    <Show>TRUE</Show>
    <Type>None</Type>
</Header>
<Header><Name>TestRequest</Name>
    <Label>TestRequest</Label>
    <Show>TRUE</Show>
    <Type>FilteredCheckList</Type>
</Header>
</SearchableHeaders>

The headers of your logged data files would contain the two channels listed above:

vsSysSpec1EngType=
TestRequest=

For more information regarding this feature, refer to LabCentral's Master Names List documentation.

Headless Mode¶

iTest can run in "headless" mode, where a GUI application (e.g., AutomationPanel) is not launched on startup. This allows for operators to perform testing on stations within a lab via LabCentral's Free-form Dashboards feature. For more information about Free-form Dashboards, refer to LabCentral's Managing Dashboards documentation.

To launch iTest in headless mode, use the following command-line argument:

iTest.exe /nogui

Notes:

  • iTest will run the solution specified in the SupportDir config.ini setting. This solution must exist in order to use the above argument.
  • LabAgent must be running on the same PC as iTest, if controlling the test station from LabCentral.
