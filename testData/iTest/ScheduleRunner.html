Overview¶

This document is to be used as reference material. For creating schedules using ScheduleRunner, refer to the Creating Schedules in Test Manager or Managing Schedules in SolutionBuilder documentation.

Previously, running a Schedule was exclusively executed by using custom VCL script. Now, a small set of standardized VCL procedures (relying on internalized functionality) manages the handling of numerous actions to support test development via the ScheduleRunner interface, including:

  • Opening and closing the schedule file.
  • Schedule validation.
  • Running each step of the schedule.
  • Handling Stop/Restart of the schedule.

iTest includes a set of VCL functions designed to take over these tasks from VCL and adds them to complied iTest code. Moving this code from script to compiled code increases performance and provides a lot more error checking. Relying on the system to execute the schedules can result in 100 Hz schedules and perhaps even faster schedules in future releases.

NOTE: This document specifically applies to schedules not prepared with BTS' PrepareCircuitSchedule.

 

VCL Commands¶

PrepareSchedule¶

The PrepareSchedule command prepares a schedule to be run by loading it into memory and performing all validation checks to verify it can run properly. If successful, the schedule is ready for RunSchedule to start execution.

Syntax:

PrepareSchedule <ID>, <Schedule.sc2>, <NumStatusChannel>, <StrStatusChannel>
PrepareScheduleEx <ID>, <Schedule.sc2>, <Page>, <NumStatusChannel>, <StrStatusChannel>

Where:

  • < ID > must be 0.
  • < Schedule.sc2 > is a literal string or a string channel containing the schedule name with the .sc2 extension.
  • < NumStatusChannel > will be set to -1 if the schedule fails to load, -2 if there is a validation error, or 1 if the schedule loads and is ready to run. If < NumStatusChannel > is negative, then < StrStatusChannel > will contain some description of the problem encountered in either loading or validating the schedule.
  • < Page > must be Data, Startup, or Shutdown.
  • Valid to PrepareSchedule at any time except when ScheduleRunner is in a running state as recorded by the vsSysSchState system channel.
NOTE: PrepareSchedule will report additional error messages with a negative error code in Softpanel; you can find the meaning using the Error Codes documentation. For example, the error code for -10 means Table error, file not found.

RunSchedule¶

The RunSchedule command starts executing a schedule previously loaded by PrepareSchedule.

Syntax:

RunSchedule <ID> <Schedule.sc2> <ReadFromFile> <Options> <Start Step> <Start Step Hook> <End Step Hook> <Status Channel>

Where:

  • < ID > must be 1. This is exclusively used only by BTS systems.
  • < Schedule.sc2 > is a literal string or a string channel containing the schedule name with the .sc2 extension.
  • < ReadFromFile > dictates the restart file to use. For example, if 1, then read restart information from the <Schedule>.1.restart file).
  • < Options > is a string of options. For example, "JTSI" has the following options:
    • J = Clears jump counters
    • T = Resets test time
    • S = Resets step time
    • I = Run immediately
  • < StartStep > is the step to start running on. If using a restart file, 0 will read the step from the file; otherwise, a positive integer will be the start step. When just starting a new test, this would be typically set to 1.
  • < Start Step Hook > is the name of a VCL procedure to execute when entering step. If you do not want to run this optional procedure, use two double-quotes (e.g., "").
  • < End Start Hook > is the name of a VCL procedure to call when leaving step. If you do not want to run this optional procedure, use two double-quotes (e.g., "").
  • < Status Channel > must be a channel, not a local variable. 0 means the schedule is not running and a positive value is the step number currently being executed. A negative status value is an error condition, and the schedule is not running. These error codes can be expanded upon, but initially, the value will be -2 = Schedule not Prepared.

Procedures called by Start Step Hook, End Step Hook, and MODE columns must execute to completion immediately. All lines must have +0.0 timestamps and no PAUSE statements. SPAWN another procedure if it is necessary to have a delay; otherwise, you will receive a warning message about how the procedure must run to completion and must not PAUSE or delay. While entering a step, a STOPSCHEDULE command in a Start Step Hook or a Mode procedure will stop the schedule before changing the setpoints but after the (internal and hidden) tablerowtodb call.

Example:

SET FromFile 1
SET StartStep 5
+0.0 RUNSCHEDULE 1 "AAAScheduleTest.sc2" FromFile "JST" StartStep "StartStepHook" "EndStepHook" numSysSchStatus

The I (uppercase letter i) option attempts to run the schedule immediately on the same scan. This is not possible when loading from a restart file, in which case the I option is ignored and a warning is generated.

CAUTION: The I option may cause scan time violations for large schedules (greater than 100 rows) and must be used with discretion.

 

Step Hooks¶

The Start Step Hook is called when entering a step after the ScheduleRunner updates channels. The End Step Hook is called when a step is over, including at the end of a test.

Step Hooks may be defined with a numeric argument to get the ID, which is useful for BTS. Do not pass the argument in the RUNSCHEDULE command. Simply define the following argument in the hook procedure (i.e., $ARGUMENT Index,Number,ByVal):

  • Name = Index
  • Type = Number
  • Pass As = Value

To guarantee real-time execution, Step Hooks are executed, not spawned, for the current scan. Any statements following a pause, delay, or switch to non-real-time will not be executed. Use extreme caution when editing step hooks; this behavior will cause subtle bugs. For example, in this hook's VCL, B's final value will be 2:

* Valid Step Hook statements because all statements execute immediately
+0.0 SET A 1
+0.0 SET B 2

+0.1 SET B 3      * Not executed because of +0.1 delay
+0.0 SET B 4      * Not executed because of preceding delay
VCLNUM Status  * Not valid, all statements must be proceeded with  +0.0

Other Schedule Control Commands¶

Schedule control can be accomplished externally via the following VCL commands and functions. Refer to the table below for a list of status channel values.

Status Channel Values

Status Channel Value Description
0 Command not complete.
1 Successful.
2 Schedule not running.

Commands:

  • ABORTSCHEDULE 1, <statuschannel> – Abort running schedule and do not save state before stopping.
  • PAUSESCHEDULE 1, <statuschannel> - Timers don't increment, ramps stop.
  • RESUMESCHEDULE 1, <statuschannel> - Compliment to PAUSESCHEDULE.
  • NEXTSTEP 1, <statuschannel> - Progress to the next step.
  • STOPSCHEDULE 1, <statuschannel> – Stop and save schedule to <schedulefile>.1.restart. Here are the contents of an example restart file (e.g., aaascheduletest.sc2.1.restart):
    Schedule=aaascheduletest.sc2
    State=Stopped
    NumRows=9
    NumCols=24
    Step=1
    StepTimeElapsed=1.500
    TestTime=21.500
    JmpCntr7=1                       
    Done=1
NOTE: JmpCntr7=1 means that the jumps done counter on Step 7 had been done 1 time.

Functions:

  • GotoStep(1, <step>) - Go to step. Returns -1 if there is an error (e.g., step out of range).
  • SaveScheduleRestart(1, <faultflag>) - Save restart information to a restart file. If faultflag is 1, save to the schedule.sc2.1.fault.restart file. Returns 1 when complete.
  • IsScheduleRunning(1) – Returns 0 or 1 if schedule 1 is running.
  • GetNumberOfSteps(1) – Returns number of steps in the schedule.
  • FreeSchedule(1, <schedule>) – Frees all memory associated with the schedule. 1 is ignored. Returns -1 of the schedule was running, 0 if schedule not prepared, 1 if a schedule was freed.

ScheduleRunner .sc2 Special Column Types, Parameters, and Validation¶

These are the column types that the internal schedule runner does something special with in addition to setting the channels (via an internal TableRowToDB).

Validation Rules:

  • Transient Tests:
    • StepTime must be a header item.
    • Limits must be a header item.
    • Limits has a limits file that exists.
    • If a test is stopped, then any log that was running on that step must also be stopped.
  • ScheduleStatusChannel header item must be numSysSchBlueBar (ScheduleStatusChannel=numSysSchBlueBar).
  • PrepareSchedule must have ScheduleRunner=Internal.
  • Special keyword in header (ScheduleRunner=Internal) means extra validation is enforced when saving schedules in Test Manager.
  • No two columns can set the same channel (except Step Tasks).
  • Check ramp by rate is always positive.
  • Check single instance of specific column types (designated below).
  • ModePair cannot be blank.
  • DataLog/LogDuration – both must be blank or both must be non-blank.
  • LogDuration cannot be 0.

Step Tasks¶

iTest schedules contain three step task columns. When a new step is entered, it is continuously evaluated in order from left to right. The first time a step task's condition evaluates as true, Schedule Runner executes the action defined in the step task. At this point, the evaluation of the step task stops, and the schedule remains on that step unless an End Step or Go To Step action is triggered.

Jump Counters¶

In non-BTS systems, the number of jumps is tracked by the channel set for the Jumps Done column. In iTest.BTS systems, the number of jumps can be tracked with a separate Virtual Output or Virtual Output Array channel. By default, iTest.BTS includes the Counter1,, Counter2, Counter3, and Counter4 virtual output channels to track the number of jumps; the channel used is specified in the Jump Setup dialog's Counter field. For more information, refer to the Jumps, Jump To, and Jump Counter Columns section in the iTest.BTS Basic Schedules documentation.

To modify the list of channels that appear in the Jump Setup dialog's Counter field, do the following:

NOTE: These steps are only relevant to iTest.BTS systems. By default, the Counter field is disabled for iTest systems.
  1. Define the virtual output channels. If BatteryAppUsesArrays=TRUE in the powertek.ini file, then these channels should be virtual output arrays.
  2. Create or edit the $SUPPORTDIR\Picklist\syscounters_h.pkl file with the list of virtual output/virtual output array channels to choose from. 
    Counter1
Counter2

Counter4
  3. Open the Jump Setup dialog in a schedule. The list of channels you can use in the Counter field is updated based upon the $SUPPORTDIR\Picklist\syscounters_h.pkl file.

Messages¶

Value Meaning
0x0 Critical errors
0x1 Errors
